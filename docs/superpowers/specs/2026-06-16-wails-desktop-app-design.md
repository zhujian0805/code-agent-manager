# Wails Desktop App for code-agent-manager — Design

**Date:** 2026-06-16
**Status:** Approved
**Owner:** code-agent-manager team

## Goal

Turn `code-agent-manager` (currently a Go CLI built on Cobra + BubbleTea) into a Wails desktop application. Expose all existing CLI operations as Wails service bindings, then build a React/TypeScript frontend that drives those bindings through a discoverable GUI. Existing CLI binaries stay fully functional.

## Non-Goals

- Removing or rewriting any existing CLI commands
- Replacing existing `internal/` business logic
- Mobile (iOS/Android) — desktop only (Linux first; macOS/Windows when possible)
- Server / web-hosted version of the GUI

### Why this structure

- The existing `internal/` packages already cleanly separate business logic from the Cobra/BubbleTea CLI layer. The Wails service layer (`internal/desktop/`) is a thin adapter — it does not re-implement anything.
- Both CLI and desktop binaries call the same Go functions and read the same config files (`~/.config/code-agent-manager/`). No state divergence is possible.
- Wails v3 auto-generates TypeScript bindings from registered Go services, eliminating manual API stubs.

## Backend — Wails Service Layer

A "service" in Wails v3 is a Go struct registered with the Wails app. Public methods are exposed to the frontend as TypeScript functions returning Promises.

### Services

| Service | Methods | Underlying packages |
|---|---|---|
| `ProviderService` | `List`, `Show`, `Add`, `Update`, `Remove`, `Enable`, `Disable`, `Rename`, `ResolveModels`, `Init` | `internal/providers` |
| `MCPService` | `ListInstalled`, `Add`, `Remove`, `SearchRegistry`, `ShowServer`, `ListClients`, `ListInstalledByClient` | `internal/mcp` |
| `EntityService` | `List(kind)`, `Search(kind, query)`, `Install(kind, ...)`, `Uninstall(kind, ...)`, `Update(kind, ...)` | `internal/entities`, `internal/repoconfig` |
| `ToolService` | `List`, `Install`, `Uninstall`, `Upgrade`, `GetLaunchPlan` | `internal/tools` |
| `DoctorService` | `RunChecks` (streams via events), `ListChecks` | `internal/doctor` |
| `ConfigService` | `ListFiles`, `Show`, `Set`, `Unset`, `Validate` (CAM + per-editor) | `internal/camconfig`, `internal/editorconfig` |
| `LaunchService` | `ListTools`, `ListProvidersForTool`, `ListModelsForProvider`, `DryRun`, `Launch` | composes `tools`, `providers` |
| `AppService` | `Version`, `OpenPath`, `OpenURL`, `Quit`, `GetTheme`, `SetTheme` | misc helpers |

### Long-running operations

Operations that may take seconds (model discovery, doctor checks, repo fetches, install commands) emit progress events via Wails events:

- `progress:<operationId>` — `{ stage: string, percent?: number, message: string }`
- `complete:<operationId>` — `{ result: any }`
- `error:<operationId>` — `{ code: string, message: string, details?: any }`

The frontend subscribes via `Events.On(...)` and shows progress in a toast/banner.

### Error contract

All service methods return Go `error`. Errors map to TypeScript Promise rejections with a structured shape:

```ts
type AppError = {
  code: string;        // e.g. "PROVIDER_NOT_FOUND"
  message: string;     // human-readable
  details?: unknown;   // structured extras (validation errors etc.)
}
```

Internally each service wraps errors with `errors.Wrap` or a small helper that produces the structured shape via Wails' error marshaling.

### Concurrency

Each service is a singleton struct with mutex-guarded state where applicable (e.g., providers file mutations). All long-running operations run in goroutines and stream via events.

## Frontend — React App

### Stack

- **Framework:** React 18 + TypeScript
- **Build:** Vite (Wails default)
- **Styling:** Tailwind CSS + Shadcn/ui (Radix primitives)
- **Routing:** React Router (sidebar nav, hash routing)
- **Server state:** TanStack Query (caching, retries, optimistic updates)
- **Client state:** Zustand (theme, app prefs)
- **Forms:** React Hook Form + Zod (provider wizard, MCP add, etc.)
- **Icons:** `lucide-react`

### Pages

1. **Dashboard / Launch** — Tool picker → provider picker → model picker, "Launch" / "Dry-run" buttons. Replaces the BubbleTea launch wizard.
2. **Providers** — Table with enable/disable toggles, "Add Provider" multi-step modal wizard, edit/remove/rename, masked API keys with reveal toggle, "Refresh models" per provider.
3. **MCP Servers** — Tabs by client (claude, cursor, gemini, etc.). Lists installed servers per client. "Add Server" modal supports custom and "From bundled registry". Registry browser with search.
4. **Library** — Tabs for Prompts / Skills / Agents / Plugins. Search bar with 3-tier search (GitHub Code Search, local store, configured repos). Install from store / GitHub / local directory. Installed list per app with uninstall multi-select.
5. **Configuration** — Tabs for CAM config + per-editor config files. JSON/TOML/YAML viewer with structured key/value editing. Scope toggle (user/project). Validation indicator.
6. **Diagnostics** — "Run all checks" button. Live-streamed pass/warn/fail with details panel per check. Re-run individual check.
7. **Settings** — App preferences (theme, log level, config dir override). About panel with version and links.

### Component library conventions

- All forms use `react-hook-form` + `zod` resolver.
- Tables use Shadcn `Table` with `tanstack/react-table` for sorting/filtering on long lists.
- Modal wizards: `Dialog` + step state machine in component-local `useReducer`.
- Toasts via Shadcn `Sonner`.

### State boundaries

- **Server data** (providers, entities, MCP servers, configs): TanStack Query keyed by service+method+args. Mutations invalidate keys.
- **UI/preferences** (theme, sidebar collapsed, last-used tool): Zustand persisted to `localStorage`.
- **In-flight wizards**: component-local state.

### Navigation

Persistent left sidebar with icons + labels. Collapsible on narrow widths (<800px window). Header shows app version + theme toggle.

## Testing Strategy

### Backend tests (Go)

- One `_test.go` per service in `internal/desktop/`.
- Use `stretchr/testify` (already a dependency).
- Each test sets up a fresh temp `CAM_CONFIG_DIR` so tests are hermetic.
- Cover happy path + at least one error path per method.
- Coverage target: **≥ 80%** for `internal/desktop/`.

### Frontend tests (TypeScript)

- **Unit tests:** Vitest + React Testing Library.
  - Components: render with mocked bindings, assert UI behavior.
  - Hooks/stores: pure logic tests.
  - Mock Wails bindings via `vi.mock('./bindings/...')` (per Wails v3 testing docs).
- **Integration tests:** Each page rendered with mocked services exercising a representative user flow (e.g., add provider → list shows it → edit → remove).
- **E2E tests:** One Playwright smoke test against a built Wails app: launch tool in dry-run mode end-to-end.
- Coverage target: **≥ 75%** for `frontend/src/`.

### CI

- Frontend: `npm run test` (Vitest + coverage), `npm run test:e2e` (Playwright, smoke only)
- Backend: `go test ./internal/desktop/... -cover`
- Build smoke: `wails3 build` on Linux runner

## Build, Packaging, Install

### Dev workflow

- `wails3 dev` — hot-reload dev server (Wails default Taskfile).
- `wails3 build` — produces `bin/cam-desktop`.
- `wails3 generate bindings` — regenerates `frontend/bindings/` from Go services.

### Distribution

- `install.sh` extended with a new `--desktop` flag:
  - default behavior unchanged (CLI install)
  - `--desktop` additionally builds and installs the Wails app:
    - binary → `~/.local/bin/cam-desktop`
    - desktop entry (Linux) → `~/.local/share/applications/cam-desktop.desktop`
    - icon (Linux) → `~/.local/share/icons/hicolor/256x256/apps/cam-desktop.png`
- Config dir (`~/.config/code-agent-manager/`) is shared with the CLI — same `providers.json`, `config.yaml`, `tools.yaml`, etc.
- `install.sh uninstall` removes both CLI binaries and (if installed) the desktop binary + desktop entry + icon.

### Platform support

- Primary: Linux (current dev environment).
- Best-effort: macOS, Windows. Wails handles cross-platform UI shell. CI runs build smoke on Linux only initially; macOS/Windows added when contributors request.

## Migration & Compatibility

- No CLI command is removed, renamed, or behavior-changed.
- Both binaries read/write the exact same files. Opening `cam-desktop` while `cam launch` is running is safe (file mutations are short and atomic).
- Wails generates TypeScript bindings; frontend never hand-writes API stubs.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Long-running ops block UI | All blocking work runs in goroutines + streamed events; UI shows progress. |
| Bindings drift from Go services | `wails3 generate bindings` runs in CI; PRs that change service signatures must include regenerated bindings. |
| Config file mutations from concurrent CLI + GUI | Serialize via the same `providers.go` mutation primitives; mutations are short. Document the shared file behavior. |
| Wails v3 still in active development | Pin a specific Wails v3 release in `go.mod` and `package.json`; bump deliberately. |
| Frontend bundle bloat | Tailwind purges; lazy-load heavy pages (Library, Configuration). |
| E2E tests flaky on CI | Keep one Playwright smoke test only; richer flows live as integration tests with mocked bindings. |

## Acceptance Criteria

- `wails3 build` produces a working `cam-desktop` binary on Linux.
- All 7 pages are functional and bound to working Go services.
- Every CLI operation has a desktop UI equivalent (or a deliberate, documented exclusion).
- `go test ./internal/desktop/...` passes with ≥ 80% coverage.
- `npm run test` passes with ≥ 75% coverage.
- One Playwright smoke E2E test passes.
- `install.sh --desktop` installs `cam-desktop`; `install.sh uninstall` cleanly removes it.
- Existing CLI tests still pass — no regressions.
